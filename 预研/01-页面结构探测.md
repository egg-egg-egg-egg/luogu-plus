## 2. ⚠️ 架构变更预警：洛谷已不再使用 `__INITIAL_STATE__`

**这是本次探测最重要的发现，请优先阅读。**

洛谷前端已从旧版（"Lentille" Vue SSR，特征是 `window.__INITIAL_STATE__ = decodeURIComponent("...")`）迁移到新版 **"Spilopelia"**（HTML `<title>` 为 `Welcome - Luogu Spilopelia`）。新架构下：

- ❌ HTML 中**不存在** `window.__INITIAL_STATE__`
- ❌ 不存在 `decodeURIComponent(...)` 包裹
- ✅ 数据载体改为 `<script id="lentille-context" type="application/json">{...}</script>`，**直接是原始 JSON**，无需 URL decode，无需 JSON.parse 时处理转义

抓取脚本 `fetch_practice.py` 内已兼容两种模式（`extract_initial_state()`），但实际命中是新架构分支。

> 对扩展的影响：原方案"从 `window.__INITIAL_STATE__` 抓 passedProblems"需改为"从 `<script id="lentille-context">` 抓 `data.passed`"。

### 2.1 数据提取正则

```python
import re, json

    html, re.DOTALL,
)
state = json.loads(m.group(1))   # 直接 JSON，无 URL decode
```

## 7. 字段映射建议（洛谷 → 我们的数据模型）

| 我们的数据模型字段 | 洛谷源字段 | 取值路径 | 备注 |
|---|---|---|---|
| `problemId`  | `pid`        | `data.passed[i].pid`        | 字符串，原样保留（P/B 前缀） |
| `title`      | `name`       | `data.passed[i].name`       | 已含 [NOIP ...] 等前缀，无需拼接 |
| `difficulty` | `difficulty` | `data.passed[i].difficulty` | number 0-7，直接保存 |
| `type`       | `type`       | `data.passed[i].type`       | "P"/"B"/… 可选字段 |
| `userId`     | `uid`        | `data.user.uid`             | number |
| `userPassedCount` | `passedProblemCount` | `data.user.passedProblemCount` | 与数组长度一致，可做校验 |

**提取算法**（伪代码）：

```ts
const m = html.match(/<script id="lentille-context" type="application\/json">([\s\S]*?)<\/script>/);
const state = JSON.parse(m[1]);
const acList = state.data.passed.map(p => ({
  problemId:  p.pid,
  title:      p.name,
  difficulty: p.difficulty,   // 0-7
  type:       p.type,         // "P" | "B" | ...
}));
// 一致性校验
console.assert(acList.length === state.data.user.passedProblemCount);
```

## 8. 风险提示

### 8.1 ✅ 未登录可拿到完整 AC 列表
本次请求**不带任何 cookie**，成功获取 `passed` 数组 206 条，与 `user.passedProblemCount=206` 完全一致。说明 practice 页面的 AC 列表对游客完全开放，扩展无需登录态即可工作。

### 8.2 ✅ 无反爬风控
HTTP 200，无登录墙、无验证码、无频率限制提示。但仍建议扩展端：
- 加 `Cache-Control` 与本地缓存（同一用户 5 分钟内不重复请求）
- 遵守 `__client_id` cookie（首次响应会下发，后续请求带上可减少风控概率）
- 批量抓取多个学生时串行 + 随机间隔（≥2s），避免触发 IP 风控

### 8.3 ⚠️ 架构可能再次变更
洛谷刚从 Lentille（`__INITIAL_STATE__`）迁到 Spilopelia（`lentille-context`），不排除后续继续调整。建议：
- 提取器做**双路兼容**：先找 `<script id="lentille-context">`，找不到再回退到 `window.__INITIAL_STATE__ = decodeURIComponent("...")`
- 提取失败时抛出明确异常，让扩展端能给老师"洛谷页面结构变更，请更新扩展"的提示

### 8.4 ⚠️ `template` 字段值与 URL 路径不一致
URL 是 `/user/{id}/practice`，但 `template` 是 `user.show`。**不要用 `template` 字段判断页面类型**，应基于 URL 路径判断。

### 8.5 ⚠️ 个别字段可能缺失
- `data.elo`：新用户 / 未参加过 ELO 比赛的用户为空数组 `[]`
- `data.user.elo`、`data.user.eloValue`、`data.user.badge`、`data.user.background`、`data.user.slogan`、`data.user.introduction`：均可为 `null` / `""`，扩展端解析时需做空值保护
- `passed` / `submitted` 元素的 `type` 字段：当前样本只有 `P`/`B`，但历史上有 `T` 等类型，不能假设枚举封闭

### 8.6 ⚠️ 题目难度可能随洛谷运营调整
`difficulty` 是洛谷官方运营标注的难度（0-7），会随题目重新评级而变化。扩展端如做历史趋势对比，应记录"抓取时刻的 difficulty"而非假定恒定。

### 8.7 ✅ 数据一致性自检字段可用
`data.user.passedProblemCount` 与 `len(data.passed)` 在本次样本完全相等（206=206）。扩展端可用此做数据完整性校验，不等则告警"洛谷返回数据不完整"。

### 8.8 ⚠️ 单页面不含每道题的 tag / 通过时间
`data.passed[i]` **只含** `type/name/difficulty/pid`，**不含**：
- 题目标签（算法分类，如 DP/图论）
- 首次 AC 时间
- AC 提交 ID

如老师需要这些信息，需额外调用 `/record/{user}` 或 `/problem/{pid}` 接口（不在本次探测范围）。
