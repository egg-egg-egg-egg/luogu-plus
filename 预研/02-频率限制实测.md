洛谷老师助手扩展中，老师会批量更新班级学生（至多 20 人）的 AC 记录，每人间隔约 2 秒。需要在**未登录状态**下实测连发请求是否触发洛谷风控，以确定扩展默认请求间隔。

### 风控判定逻辑

每次请求记录：HTTP 状态码、响应大小、请求耗时、是否重定向、`<title>`、`__INITIAL_STATE__` 是否存在、风控关键词命中、关键响应头。

判定为触发风控的条件（满足任一）：
1. 状态码 ≠ 200
2. 请求被重定向到非目标 URL
3. 命中强风控关键词（验证 / block / 拒绝 / 频繁 / 稍后 / 限制 / captcha / 禁止 / forbidden / rate limit）
4. 响应大小偏离基线 > 30%（疑似被替换为拦截/验证页）

> **重要校准**：经基线探测，匿名访问 `user/{uid}/practice` 返回的是 SPA 外壳 HTML（title 为 "Welcome - Luogu Spilopelia"），**不含 `__INITIAL_STATE__`**（用户数据由前端 SPA 路由 + AJAX 加载）。因此 `__INITIAL_STATE__` 缺失在匿名态下是**正常基线**，不作为风控触发条件，仅作数据记录。这与登录态页面不同（登录态页面会在 HTML 内注入 `__INITIAL_STATE__`）。

## 四、结论

### 1. 是否触发风控

| 轮次 | 间隔 | 触发风控 | 触发时机 | 恢复时间 |
|------|------|----------|----------|----------|
| 第一轮 | 2s | **否** | — | — |
| 第二轮 | 3s | **否** | — | — |

**两轮共 40 次请求（外加 1 次基线探测），全部返回 200，无任何风控特征。** 2 秒间隔连发 20 次未登录请求，未触发洛谷风控。

### 2. 风控特征（本次未观测到，以下为预期特征供扩展实现监控）

本次测试未触发风控，因此无法给出实测的风控响应样本。根据洛谷架构（自建 CDN + 多 web 节点 + CSP 中引用 Cloudflare Challenge 与网易易盾验证码），**若触发风控，预期可能表现为**：

- **状态码**：403 / 429 / 503（非 200）
- **响应内容**：含"验证"/"频繁"/"稍后"/"block"等关键词；或重定向到验证码页（网易易盾 `*.dun.163yun.com`）/ Cloudflare Challenge 页
- **响应大小**：明显偏离基线 24228 字节（拦截页通常更小）
- **响应头**：可能出现 `Retry-After`、`Location`（重定向）
- **`<title>`**：不再是 "Welcome - Luogu Spilopelia"

扩展实现时建议监控上述信号，一旦命中即进入退避重试。

### 3. 推荐默认间隔：**2 秒**

**依据**：
1. 实测 2s 间隔连发 20 次（覆盖老师批量刷新 20 个学生的上限场景）**未触发任何风控**，全部 200 且响应稳定。
2. 响应耗时稳定（0.28–0.36s），无随请求次数递增的退化趋势，说明源站与 CDN 均未施加速率压制。
3. 3s 间隔同样安全，但 2s 已足够安全，且能更快完成批量刷新（20 人 × 2s ≈ 40s vs 3s ≈ 60s）。

**保留 3s 作为保守降级档**：若扩展上线后监控到偶发 429/风控（例如高峰期、或登录态下接口限流更严），可一键切换到 3s。5s 在本次测试中无必要，过于保守会拖慢老师操作。

### 4. 风控恢复策略建议

虽未实测到风控，但基于洛谷架构给出建议（供扩展实现）：

1. **指数退避重试**：遇风控（非 200 / 关键词 / 重定向）时，等待 30s → 60s → 120s 递增重试，最多 3 次。
2. **触发后冷却**：单次风控触发后，对后续请求间隔自动翻倍（2s → 4s）并持续 5 分钟，无再次触发再回落。
3. **5 分钟恢复窗口**：洛谷 `C3VK` cookie 有效期为 5 分钟，推测其短时频控窗口在分钟级；触发后等待 5 分钟再重试是较稳妥的恢复策略。
4. **失败兜底**：连续 3 次重试仍风控，则停止本轮批量刷新，提示老师稍后手动重试，避免被进一步升级封禁。
5. **登录态注意**：本次为未登录测试；扩展实际使用老师登录态，登录态下接口（尤其取 AC 记录的 API 端点）可能有独立的频控，建议上线后对真实接口再做一次登录态压测。

## 五、测试局限说明

1. **同 URL 重复请求**：本次为控制变量，对同一用户页面重复请求 20 次（CDN 命中率高）。实际场景是 20 个**不同**学生页面，CDN 命中率较低、源站压力略高，风控阈值可能不同。建议后续补充"20 个不同 UID"的对比测试。
2. **未登录态**：本次测试未登录。扩展实际使用老师登录态，登录态的频控策略可能不同（可能更严，因可识别用户）。
3. **HTML 端点 vs API 端点**：本次测的是 HTML 页面 `user/{uid}/practice`；扩展若改用 JSON API（如 `user/{uid}?_contentOnly=1`）取数据，该端点频控可能不同。
4. **单次测试**：风控可能与时段、源站负载相关，单次测试通过不代表所有时段均安全。
5. **测试环境**：从本机出口 IP 发起，若扩展用户与测试机出口 IP 段不同，结果可能有差异。
