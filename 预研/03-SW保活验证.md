# 03 — MV3 Service Worker 保活原型验证

> 目标：验证 `chrome.alarms` 心跳保活 + `chrome.storage.local` 断点续传方案，能否让 MV3 Service Worker 跑完一个 40 秒的长任务（班级批量更新 20 名学生 AC 记录，每 2 秒一个）。

## 一、问题背景

MV3 的 Service Worker（下称 SW）默认 **30 秒无活动即休眠**。班级批量更新场景：20 人 × 2s ≈ 40s，超过 30s 阈值，任务会被中途 kill，导致进度丢失、用户重试。

需要一个保活 + 可恢复的方案。本原型验证最轻量的组合：**alarms 心跳 + storage 断点续传**。

## 二、原型架构

四个角色协作：

```
┌─────────────────────────────────────────────────────────────┐
│  popup.html / popup.js  (UI 层，可随时关闭)                  │
│   - 启动任务按钮 / 重置按钮                                   │
│   - 每 500ms 向 SW 轮询 GET_STATUS 刷新进度条与日志          │
│   - 关闭 popup 不影响后台任务                                 │
└───────────────┬─────────────────────────────────────────────┘
                │ chrome.runtime.sendMessage
                ▼
┌─────────────────────────────────────────────────────────────┐
│  background.js  (Service Worker, ES module)                 │
│                                                             │
│  ┌─────────────── 长任务循环 ───────────────┐               │
│  │ setInterval(tick, 2000)                   │               │
│  │   tick: 处理 1 名学生 → 写 storage.local  │               │
│  │   完成后 clear alarm + status=completed   │               │
│  └───────────────────────────────────────────┘               │
│                                                             │
│  ┌─────────────── 心跳保活 ─────────────────┐               │
│  │ chrome.alarms.create('keepalive', 0.4min)│               │
│  │ onAlarm → getPlatformInfo() 续命          │               │
│  └───────────────────────────────────────────┘               │
│                                                             │
│  ┌─────────── SW 启动恢复 (顶层 bootstrap) ──┐              │
│  │ 读取 storage.local                        │              │
│  │ 若 status='running' → resumeTask()        │              │
│  └───────────────────────────────────────────┘              │
└───────────────┬───────────────────────────┬─────────────────┘
                │                           │
        chrome.alarms                chrome.storage.local
                │                           │
                ▼                           ▼
        ┌───────────────┐         ┌────────────────────┐
        │ Chrome 调度器  │         │ 持久化进度存档      │
        │ 每 24s 唤醒 SW │         │ {taskId,done,...}   │
        └───────────────┘         └────────────────────┘
```

**关键交互链路：**

1. **popup → SW**：`START_TASK` 启动；`GET_STATUS` 轮询；`RESET` 重置。
2. **SW → storage.local**：每次 `tick` 写进度（done / students / logs）。这是断点续传的"存档点"。
3. **alarms → SW**：每 24s 唤醒 SW 并调用 `getPlatformInfo()`，重置 30s 空闲计时器。
4. **SW 重启 → storage.local → 恢复**：SW 被杀后，下次被任意事件（alarm / 消息 / 点击图标）唤醒时，顶层 `bootstrap()` 读取存档并从 `done` 续传。

## 三、核心机制说明

### 3.1 心跳保活（alarms + API 调用）

```js
// background.js
chrome.alarms.create(ALARM_NAME, { periodInMinutes: 0.4 }); // 24s

chrome.alarms.onAlarm.addListener(async (alarm) => {
  if (alarm.name !== ALARM_NAME) return;
  await chrome.runtime.getPlatformInfo(); // 关键：异步 API 调用给 SW 续命
});
```

**原理**：`setInterval` 本身不能阻止 SW 休眠。Chrome 在"事件回调中正在处理 / 有 pending 的异步 API 调用"时才延长 SW 寿命。alarm 每 24s 触发一次事件，回调里调用 `getPlatformInfo()` 制造一次 pending API 调用，从而把 30s 空闲计时器重置。24s < 30s，留 6s 余量。

### 3.2 断点续传（storage.local 存档）

```js
// 每次 tick 写进度
await writeState({
  done: studentNo,
  students: [...(state.students || []), student],
  logs: [...]
});
```

**存档结构**：

```js
{
  taskId: 'task_1735680000000',
  total: 20,
  done: 7,              // 已完成 7 个
  status: 'running',
  startedAt: '...',
  students: [{ index: 1, name: '学生1', finishedAt: '...' }, ...],
  logs: ['[HH:MM:SS] 处理学生 3/20 完成', ...]
}
```

**以 storage.done 为唯一事实源**：`tick` 每次先读 `done` 再算下一个，避免内存/存储不一致导致重复处理。

### 3.3 SW 重启恢复（顶层 bootstrap）

```js
// SW 每次启动都跑顶层代码
(async function bootstrap() {
  const state = await readState();
  if (state && state.status === 'running') {
    await resumeTask(state); // 重建 alarm + 重启 setInterval
  }
})();
```

**双保险**：即便心跳没能拦住休眠（例如 Chrome 钳制 alarm 到 1 分钟），SW 被杀后下一次被唤醒时，`bootstrap()` 自动从 `done` 续传，任务最终必然完成。代价是中间会有"停顿—恢复"的间隙，但不会丢进度。

> 边界说明：若 SW 在"已处理学生 N"但"尚未写回 done=N"之间被杀，恢复后会从 done=N-1 重做学生 N。真实业务里抓取接口需做成幂等（或用 students 数组去重）。

## 四、手动验证步骤

> 环境：Chrome 最新稳定版，加载"已解压的扩展程序"。

### 步骤 1：加载扩展

1. 打开 `chrome://extensions`
2. 右上角开启「开发者模式」
3. 点「加载已解压的扩展程序」，选择目录：
   `D:/Workspace/workbuddy/luogu-plus/预研/sw-keepalive-prototype`
4. 确认扩展出现在列表中，且 Service Worker 显示为 "active" / "inactive"

### 步骤 2：验证保活（关闭 popup 后任务继续）

1. 点击扩展图标打开 popup
2. 点「启动任务」按钮
3. 确认进度条开始走、日志区开始滚动（`处理学生 1/20 完成` ...）
4. **立即关闭 popup**（点窗口外或关闭按钮）
5. 等 30 秒后，回到 `chrome://extensions`，观察该扩展的 Service Worker 状态：
   - 期望：仍为 **active**（或周期性 active），未因空闲被 kill
6. 等任务总时长到约 40 秒后，重新点击扩展图标打开 popup：
   - 期望：进度到 **20 / 20**，状态显示 **completed**，日志最后一条为「任务完成」

### 步骤 3：验证断点续传（手动 kill SW）

1. 先点「重置」回到 idle
2. 打开 popup，点「启动任务」
3. 等进度走到 3~5 / 20 之间
4. 到 `chrome://extensions`，找到该扩展，点 Service Worker 旁边的「终止 / Terminate」按钮（手动模拟 SW 被杀）
5. 观察：扩展图标旁的 SW 状态变成 inactive
6. 重新点击扩展图标打开 popup（这会通过消息事件唤醒 SW）：
   - 期望：日志区出现一条 `[HH:MM:SS] SW 重启，从 N/20 断点续传`
   - 期望：进度从 N 继续往上走，最终到 20 / 20 completed

### 步骤 4：验证完全无 popup 的后台恢复

1. 重置后点「启动任务」，立即关闭 popup
2. 不再点扩展图标（不主动唤醒）
3. 等 40 秒以上，打开 `chrome://extensions` → 点该扩展的「Service Worker」链接查看 Console
4. 期望：Console 可见 bootstrap 恢复日志（若 SW 中途被杀过）或任务已 completed

## 六、备选方案：Offscreen Document

若验证发现 alarms 心跳在目标 Chrome 版本下不可靠（例如生产环境 alarm 被钳制到 1 分钟，且 SW 频繁被杀导致任务停顿明显），则切换到 **Offscreen Document** 方案。

### 原理

MV3 允许扩展创建一个隐藏的 DOM 文档（offscreen document），它有独立于 SW 的生命周期，只要文档存在就持续运行。把长任务的 `setInterval` 放到 offscreen document 里执行，进度通过 `chrome.runtime.sendMessage` 回传给 SW / popup。

- SW 只做"创建 offscreen document + 转发消息"
- offscreen document 跑真正的循环（普通 DOM 环境，定时器不会被 SW 休眠打断）
- 进度仍写 `chrome.storage.local`，popup 轮询不变

### 切换成本

| 维度 | 成本 |
| --- | --- |
| 权限 | manifest 需新增 `"offscreen"` permission |
| 代码 | 新增 `offscreen.html` + `offscreen.js`；`background.js` 的 `tick` 循环迁移过去 |
| 复杂度 | 多一层消息中转（SW ↔ offscreen ↔ popup），调试更麻烦 |
| 生命周期 | 需在任务结束时 `chrome.offscreen.closeDocument()`，否则泄漏 |
| 兼容性 | Offscreen API 需 Chrome 109+，目前主流版本均满足 |

**决策建议**：优先用本原型的 alarms 方案（更轻、代码量少、可直接搬入主项目）。仅当步骤 2/4 验证失败或生产环境稳定性不达标时，再切 Offscreen Document。两套方案的 `storage.local` 进度结构与 `tick` 逻辑可完全复用，切换时只换"循环宿主"。

## 七、代码复用说明（搬入主项目 `src/background/`）

以下代码可**直接**迁移到主项目，仅需替换模拟抓取为真实洛谷请求：

| 原型位置 | 主项目建议位置 | 迁移要点 |
| --- | --- | --- |
| `manifest.json` 的 alarms/storage 权限 | `src/manifest.json` | 合并进现有 permissions；`background.service_worker` 指向主项目打包产物 |
| `background.js` 常量段（STORAGE_KEY / ALARM_* / LOOP_* / TOTAL_*） | `src/background/config.js` | LOOP_INTERVAL 与 TOTAL 改为按班级动态计算 |
| `readState` / `writeState` / `stopLoop` | `src/background/task-store.js` | 类型化（迁移 TS 时加 `TaskState` interface） |
| `startTask` / `resumeTask` / `tick` / `resetTask` | `src/background/batch-task.js` | `tick` 内的"模拟处理"替换为 `fetchStudentAC(uid)` 真实抓取 + 解析 |
| `chrome.alarms.onAlarm` 保活段 | `src/background/keepalive.js` | 保持不变，是通用保活模块 |
| `bootstrap()` 顶层恢复 | `src/background/index.js` 顶层 | 放在 SW 入口最顶层，确保任何唤醒路径都先跑恢复 |
| `chrome.runtime.onMessage` 监听 | `src/background/messaging.js` | 增加 `ABORT_TASK` 等业务消息 |

**迁移到 TypeScript 注意**：
- 给 `TaskState`、`Student`、`PopupMessage`、`BgResponse` 定义 interface
- `chrome` 全局用 `@types/chrome`
- 顶层 `bootstrap` IIFE 保持纯 JS 风格也可，TS 下用 `(async () => { ... })()`

**真实业务替换点**（`tick` 内）：

```ts
// 原型（模拟）
const student = { index: studentNo, name: `学生${studentNo}`, finishedAt: new Date().toISOString() };

// 主项目（真实抓取）
const ac = await fetchStudentAC(student.uid);   // 抓洛谷用户页解析 AC 数
await updateStudentRecord(student.uid, ac);      // 落库
const student = { index: studentNo, uid: student.uid, ac, finishedAt: new Date().toISOString() };
```

其余保活、断点续传、恢复逻辑**零改动**复用。
