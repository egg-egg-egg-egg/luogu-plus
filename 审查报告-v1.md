> 审查人：subagent（资深 PM + 浏览器扩展架构师视角）

## 1. 需求完整性

### P0 — 学生删除后的级联清理未定义
- **位置**：3.1「编辑-支持改备注、删除学生」+ 5.2 班级表 `studentIds[]`
- **问题**：删除学生时，`acRecords` 表里的记录是否清理？所有班级的 `studentIds[]` 数组如何同步移除？从班级移除学生 vs 从系统彻底删除，两种操作的语义和数据后果完全没区分。
- **建议**：明确两种操作——「从班级移除」（保留 student 实体和 acRecords，仅断开班级关系）vs「彻底删除学生」（级联清理 acRecords + 遍历所有 classes 的 studentIds）。建议彻底删除走二次确认 + 软删除（标记 `deletedAt`），7 天后才物理删，防误操作。

### P0 — 学生洛谷昵称变更无刷新机制
- **位置**：3.1「luoguName（自动填充）」
- **问题**：学生改了洛谷昵称后，老师看到的永远是建档时的旧昵称。文档没有任何机制触发昵称刷新。
- **建议**：每次「更新该学生 AC 记录」时顺带刷新 `luoguName`；管理页学生列表加一个「刷新昵称」按钮；如果 `luoguName` 与最近一次抓取的不一致，UI 上提示「昵称已变化」。

### P0 — 班级删除后学生 AC 缓存策略未定义
- **位置**：3.2「操作-删除」
- **问题**：删班级时，里面学生的 acRecords 留不留？如果一个学生只在这个班级，删完班级后这堆 acRecords 成了孤儿数据。
- **建议**：删班级只删 classes 表记录，不动 students 和 acRecords（学生可能还在别的班）。但要补一个「孤儿清理」工具：删除学生时才级联删 acRecords。

### P0 — 班级批量更新中部分失败的处理未定义
- **位置**：6「失败重试-退避 10 秒，最多重试 2 次」
- **问题**：20 人班级更新，第 7 个学生重试 2 次还失败，后续 13 人是继续还是终止？已成功的 6 人数据是保留还是回滚？老师怎么知道是哪个学生失败？
- **建议**：失败不影响后续（best-effort 模式），保留已成功数据；更新结束后 UI 列出失败学生清单 + 单独重试按钮；进度条显示「17/20 成功，3 失败」。

### P0 — 缓存"过期"感知完全缺失
- **位置**：3.3 + 第 5 章数据模型
- **问题**：老师可能忘了上次更新是什么时候。文档没有任何 `lastSyncedAt` 字段，老师面对徽章无法判断数据新鲜度。
- **建议**：Student 和 Class 都加 `lastSyncedAt: number | null`；徽章 tooltip 显示「数据更新于 3 天前」；超过 N 天徽章颜色变浅或加时钟图标。

### P1 — 重复学生 / 同一班级重复添加未定义
- **建议**：luoguId 全局唯一索引，重复添加报错；班级的 studentIds 入班时去重。

### P1 — 批量导入流程未定义
- **建议**：补 3.1.x 小节定义批量导入：输入格式、单行校验、串行抓取昵称（间隔 2s）、失败行高亮可重试。

### P1 — 空班级 / 0 AC 学生的边缘情况未提
- **建议**：空班级更新按钮置灰；0 AC 学生正常入库，acRecords 表对应 0 条记录。

### P1 — 学生洛谷账号被封禁/注销的处理
- **建议**：抓取时区分 HTTP 状态——404 标记 `status = 'invalid'` UI 红字提示；403 同上；5xx 不更新本地数据，提示稍后重试。

### P1 — 洛谷题目难度会变化，缓存如何同步
- **建议**：difficulty 单独建 `problems` 表（题目主键 + 最新难度 + lastUpdatedAt），acRecords 只存 problemId 引用。每次抓取学生时顺带刷新该题难度。

### P2 — 国际化预留
- **建议**：v1 不做翻译，但代码用 `t('key')` 模式，文案集中管理。

## 2. 技术可行性

### P0 — MV3 Service Worker 休眠风险
- **位置**：第 4 章 + 第 6 章「班级更新 20 人 × 2s = 40s」
- **问题**：Chrome MV3 的 service worker 默认 30 秒无活动就休眠。班级更新 40 秒的长任务必然被中断。这是整个项目最大的技术风险。
- **建议**：三种保活方案任选其一：
  1. **chrome.alarms 心跳保活**：每 25 秒触发 alarm 唤醒 SW（推荐，简单）
  2. **可恢复任务设计**：每抓完一个学生写 IndexedDB 进度，SW 被杀后从断点续传
  3. **Offscreen Document**：把抓取逻辑放到 offscreen document 里跑

### P0 — 抓取架构（content script vs background fetch）未决策
- **位置**：3.3 + 第 4 章
- **问题**：20 人班级批量更新，不可能开 20 个隐藏 tab 让 content script 抓。必然是 background 发 fetch 拿 HTML + DOM 解析。但文档没说这个架构决策，也没说用什么库解析。另外，practice 页面是 Vue SSR，HTML 里嵌的 `window.__INITIAL_STATE__` JSON 才是稳定数据源，DOM 反而不稳。
- **建议**：明确——background script 用 `fetch()` 拉 HTML → 提取 `__INITIAL_STATE__` JSON → 拿到结构化的 `passedProblems` 数组。**不要解析 DOM**。

### P0 — 难度标签抓取策略与数据源矛盾
- **位置**：3.3「抓取内容-AC 题目ID + 题目难度标签」
- **问题**：洛谷 `/user/{id}/practice` 页面的 `__INITIAL_STATE__` 里每道 AC 题目带 `difficulty` 字段（0-7 数字映射），技术上可行。但文档没说难度是从 practice 页面拿还是从每道题详情页拿——如果是后者，请求量爆炸。
- **建议**：明确「难度标签从 practice 页面 `__INITIAL_STATE__.passedProblems[].difficulty` 一次性获取」。洛谷难度映射：0=暂无评定，1=入门，2=普及-，3=普及/提高-，4=普及+/提高，5=提高+/省选-，6=省选/NOI-，7=NOI/NOI+。

### P0 — 列表页选择器适配多页面结构未规划
- **位置**：3.4.2
- **问题**：三个页面 DOM 结构完全不同：`/training/{id}` 是卡片网格，`/problem/list` 是表格行，`/problemset` 也是表格但列结构不同。文档统一说「每道题卡片右侧注入徽章」——但 `/problem/list` 根本没有卡片！事实性错误。另外，洛谷是 Vue SPA，路由切换不会重新加载页面，content script 怎么感知路由变化？
- **建议**：分页面定义选择器策略——
  - `/training/{id}`：`.training-problem-card` 卡片右下角注入
  - `/problem/list`：`tr[data-pid]` 行末尾追加 `<td>`
  - `/problemset`：同上但选择器不同
  路由监听用 `MutationObserver` 监听 `body` 子树 + URL 轮询双保险。

### P0 — IndexedDB Schema 升级路径未规划
- **建议**：Dexie schema 版本从 `v1` 起步，规定「字段新增必须升版本号 + 写 upgrade 函数」。

### P1 — Zustand 跨上下文通信"友好"是误导
- **问题**：Zustand 本身不跨 content/popup/background 上下文。三个上下文各有自己的 JS 运行时，store 是独立的。
- **建议**：改为「Zustand 管理各上下文内 UI 状态，跨上下文同步通过 `chrome.storage.local` + `chrome.storage.onChanged` 实现，封装 `useSyncedStore` hook」。

### P1 — Content Script 注入时机与 SPA 路由未规划
- **建议**：用 `MutationObserver` 监听 `document.body` + URL 变化，封装 `onRouteChange(callback)` 工具。

### P1 — 洛谷反爬验证页识别未定义
- **建议**：抓取后校验响应——检测 HTML 是否包含「登录」「验证」「block」关键词，或检测 `__INITIAL_STATE__` 是否存在。命中则标记「触发风控，暂停 5 分钟」。

### P2 — 包体预估 1.5MB 偏乐观
- **建议**：补依赖体积预算表，发版前用 `vite-bundle-visualizer` 验证。

## 3. 矛盾与歧义

### P0 — 「每道题卡片右侧注入徽章」与 `/problem/list` 表格结构矛盾
- **建议**：改为「按各页面原生结构适配——卡片页注入到卡片内，表格页追加一列」。

### P1 — "会话"定义不清
- **位置**：3.4.3「状态条可关闭（本次会话不再显示）」
- **建议**：改为「关闭后，当前题目页面本次浏览期间不再显示，刷新或重新打开恢复显示」。或干脆改成「关闭后该题目永久不显示，管理页可恢复」。

### P1 — Popup vs Side Panel 职责边界未明确
- **问题**：3.4.1 同时提到 Popup 和 Side Panel，但 3.5 只描述 Popup。到底是同时支持两个，还是二选一？
- **建议**：v1 只做 Popup + 管理页，明确 Side Panel 列入 v2。**必须二选一**。

### P1 — `currentSelection` 存储位置与持久化语义不清
- **问题**：`currentSelection` 跨标签页共享吗？老师在 A 标签页选了班级，B 标签页的洛谷题目列表会立即跟着标注吗？
- **建议**：`currentSelection` 全局唯一，存 `chrome.storage.local`，通过 `onChanged` 事件广播到所有标签页。

### P1 — 多对多关系用数组存储的维护成本未说明
- **建议**：改为关系表 `class_members(classId, studentId)`，符合关系型思维。

### P1 — 「不需要登录态」未经验证
- **建议**：开发前置项加「洛谷未登录访问探测」任务——清缓存无 cookie 访问 `/user/1403058/practice`，验证 ①能否拿到完整 AC 列表 ②是否有频率限制 ③`__INITIAL_STATE__` 是否完整。

### P2 — 设置页可调间隔与默认值散落多处
- **建议**：单独抽一个「默认配置表」章节。

## 4. 用户体验

### P0 — 班级更新 40 秒的进度反馈与离开恢复未定义
- **建议**：①更新任务跑在 background，popup 关闭不影响；②popup 重新打开时读取 background 任务状态恢复进度；③完成后发 `chrome.notifications`；④管理页班级列表显示「更新中 5/20」。

### P0 — 选择状态跨标签页 / 跨会话持久化未定义
- **建议**：`currentSelection` 持久化到 `chrome.storage.local`，跨标签页跨会话保持。浏览器重启后默认恢复上次选择。

### P1 — 缓存过期感知缺失（用户体验维度）
- **建议**：徽章 hover tooltip 显示「数据更新于 2026-06-15（18 天前）」；超过 7 天徽章加褪色效果。

### P1 — 多标签页同时打开洛谷的徽章重复注入风险
- **建议**：注入前 `querySelector('[data-luogu-plus-badge]')` 检查是否已注入。

### P1 — 学生添加时昵称拉取的 UI 反馈未定义
- **建议**：输入 ID 实时校验 → 点「添加」变 loading + 禁用 → 成功后学生卡片淡入 → 失败 toast 报错 + 输入框保留待重试。

### P2 — 「玻璃质感小徽章」无视觉规格
- **建议**：给出具体规格——`padding: 2px 8px; border-radius: 4px; font-size: 12px; ...`。

### P2 — 暗色模式下徽章可读性未提
- **建议**：明暗模式各一套色板。

## 5. 安全与合规

### P0 — 洛谷 ToS / robots.txt 合规性未评估
- **问题**：洛谷用户协议是否禁止自动化抓取？`robots.txt` 对 `/user/*/practice` 的态度？这是项目能否上架的法律前提。
- **建议**：开发前置项加「合规评估」任务——①抓 `robots.txt` 确认 Disallow；②阅读用户协议；③如果禁止，评估灰色地带。**结论写入规格附录**。

### P0 — 扩展商店上架审核风险未评估
- **建议**：补「上架准备」章节——①撰写隐私政策；②`host_permissions` 仅申请 `*://www.luogu.com.cn/*`；③准备审核说明文档；④准备被拒后申诉材料。

### P1 — 未登录高频请求的封 IP 风险
- **建议**：①开发期实测清 cookie 后连发 20 次请求；②如果触发风控，默认间隔提到 3-5 秒；③加风控检测逻辑；④配合 Cache-Control 头。

### P1 — 学生数据隐私合规（PIPL / 未成年人）
- **建议**：①扩展卸载时 IndexedDB 自动清除（写进隐私政策）；②学生备注名允许留空或化名；③隐私政策明确「数据仅本地」。

### P2 — Manifest 权限最小化未列
- **建议**：权限清单——`storage`、`notifications`、`host_permissions: *://www.luogu.com.cn/*`。不要申请 `tabs`、`<all_urls>`。

## 6. 文档质量

### P0 — 数据模型缺 `lastSyncedAt` 字段
- **建议**：Student 和 Class 都加 `lastSyncedAt: number | null`。Class 取班级内所有学生 `lastSyncedAt` 的最小值。

### P0 — 测试策略完全缺失
- **建议**：补「测试策略」章节——①单元测试：抓取解析函数用录制 fixture 测试；②集成测试：Dexie schema 升级；③E2E：Playwright 驱动加载扩展访问 mock 洛谷页面；④回归测试：洛谷改版后跑 fixture 测试。

### P0 — 验收标准缺失
- **建议**：核心功能补验收标准——①班级更新：20 人 ≤ 60 秒，失败学生清晰列出；②徽章注入：页面加载后 500ms 内出现；③缓存查询：单题 ≤ 10ms；④内存占用：< 50MB。

### P1 — 错误码 / 错误信息规范未定义
- **建议**：定义错误码表——`E_LUOGU_404`、`E_LUOGU_403`、`E_LUOGU_RATE_LIMIT`、`E_NETWORK`、`E_PARSE`。每个错误码对应用户友好文案 + 建议操作。

### P1 — 性能指标未量化
- **建议**：补「性能基线」表。

### P2 — 日志与错误上报未提
- **建议**：v1 做本地日志，管理页有「导出诊断日志」按钮。不上报远程。

### P2 — 无障碍（a11y）未提
- **建议**：徽章加 `aria-label`，状态条加 `role="status" aria-live="polite"`。

### P2 — 浏览器兼容范围说明不足
- **建议**：明确「v1 仅支持 Chromium 系（Chrome/Edge 114+），Firefox/Safari 不支持」。

## 总体评价

**结论：不能直接进入开发。至少需要再补一轮规格迭代。**

存在 **6 个 P0 级阻塞问题**：
1. MV3 Service Worker 休眠——40 秒长任务必然中断，保活方案必须先选定
2. 抓取架构未决策——background fetch + `__INITIAL_STATE__` 解析这条路径必须写死
3. 难度标签数据源含糊——必须明确从 practice 页面一次拿全
4. 列表页选择器适配——「卡片右侧注入」与表格页结构矛盾
5. 缓存过期感知缺失——`lastSyncedAt` 字段缺失影响核心 UX
6. 合规与上架风险零评估——洛谷 ToS 和 Chrome Web Store 审核是法律前提

另有 ~15 个 P1 问题建议在开发启动前补齐。

**建议路径**：
- 用 1-2 天做 3 个预研任务：①洛谷 practice 页面 `__INITIAL_STATE__` 结构探测；②未登录访问频率限制实测；③MV3 SW 保活方案原型验证
- 预研结果回填规格，补写「合规评估」「测试策略」「验收标准」「错误码表」四个章节
- 数据模型加 `lastSyncedAt`，明确 schema 升级路径
- 再走一次评审，确认无 P0 后开工
