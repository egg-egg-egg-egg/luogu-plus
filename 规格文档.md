## 1. 项目概述

### 1.1 一句话定位
洛谷浏览器扩展，帮信息学老师避开"布置到学生已经 AC 的题"的痛点。

### 1.2 核心痛点
老师出题时，要在洛谷题库里挑题。但学生可能已经 AC 过这些题——布置了等于白布置。手动挨个学生去翻 `/user/{id}/practice` 页面太慢，几十个学生根本翻不过来。

### 1.3 解决思路
- 老师在插件里维护学生和班级
- 一键拉取学生 AC 记录（含难度标签）到本地缓存
- 老师在洛谷题目列表/详情页选定学生或班级后，插件实时标注哪些题"已被 AC"，并显示数据新鲜度

### 1.4 分发方式
**开源项目 + 老师内部分发**，不上架 Chrome Web Store。绕开商店审核合规要求，但仍需遵守洛谷 ToS（见 §12）。

## 2. 用户场景

### 场景 A：学期初建档
老师新建班级"提高班 2026 春"，把学生按洛谷ID添加进来，给每人填上备注名（真实姓名）。点"批量更新"，等约 40 秒，所有人的 AC 记录就缓存到本地了。徽章 tooltip 显示"数据更新于刚刚"。

### 场景 B：出题查重
老师打开题单页 `/training/xxx`，在 Popup 里选"提高班 2026 春"，题单每道题旁边立即显示徽章「✅ 3/20 已AC」。鼠标悬停徽章，看到是张三、李四、王五做的，以及"数据更新于 2 天前"。

### 场景 C：题目确认
老师点开一道题 `/problem/P1234`，顶部出现状态条「本班 5/20 已AC：张三、李四、王五、赵六、孙七」。如果只点了单个学生，状态条变成「张三 已 AC」或「张三 未做过」。

### 场景 D：按需更新
老师在班级列表里点"更新"按钮（可选择更新某个学生或更新整个班级），插件重新拉取该目标的最新 AC 记录，覆盖本地缓存。更新频率完全由老师决定，无固定周期。徽章 tooltip 的"数据更新于"会同步刷新。

### 场景 E：识别过期数据
老师打开题单，发现徽章颜色偏淡且带时钟图标。hover 后看到"数据更新于 12 天前，建议刷新"。老师点更新按钮，徽章恢复正常颜色。

## 3. 功能规格

### 3.1 学生管理
| 项 | 说明 |
|---|---|
| 添加方式 | 输入洛谷ID，插件自动拉取洛谷昵称 |
| 字段 | `luoguId`（必填，全局唯一）、`luoguName`（自动填充，更新时顺带刷新）、`remark`（老师备注，真实姓名，可留空）、`status`（`active` / `invalid`）、`lastSyncedAt`（最近一次成功更新 AC 记录的时间戳，`null` 表示从未更新） |
| 上限 | 无硬性上限（实际场景通常 ≤ 70，不强制） |
| 编辑 | 支持改备注、删除学生 |
| 删除语义 | 二次确认 → 硬删 → 级联清理：①该学生的 `acRecords` ②`class_members` 关系表中的引用 ③`problems` 表不动（题目本身可能被其他学生 AC 过） |
| 校验 | 洛谷ID 必须数字；添加时验证昵称能否拉到（拉不到标记 `status='invalid'` 并报错）；luoguId 全局唯一，重复添加报错"该学生已存在" |
| 昵称刷新 | 每次"更新该学生 AC 记录"时顺带用最新抓取的昵称覆盖 `luoguName`；管理页学生列表加单独"刷新昵称"按钮 |
| 状态失效 | 抓取时遇 404/403 → 标记 `status='invalid'`，UI 红字提示"该洛谷账号可能已注销/封禁" |

### 3.2 班级管理
| 项 | 说明 |
|---|---|
| 关系 | 学生 ↔ 班级 = **多对多**，通过 `class_members` 关系表实现（见 §5.4） |
| 字段 | `id`、`name`、`createdAt`、`updatedAt`、`lastSyncedAt`（班级内所有学生 `lastSyncedAt` 的最小值，`null` 表示有人从未更新） |
| 上限 | 无硬性上限（实际场景通常 ≤ 20，不强制） |
| 操作 | 创建、重命名、删除（仅删 `classes` 记录 + `class_members` 关系，**不动** `students` 和 `acRecords`——学生可能在别的班） |
| 重复添加 | 同一学生加入同一班级自动去重，不报错（幂等） |
| 空班级 | 允许存在；空班级的"更新"按钮置灰 |

### 3.3 做题记录查询
| 项 | 说明 |
|---|---|
| 数据源 | 洛谷 `https://www.luogu.com.cn/user/{id}/practice` 页面 |
| 抓取架构 | **background service worker** 发 `fetch()` 拉 HTML → 提取 `<script id="lentille-context" type="application/json">` 内的 JSON → 解析 `data.passed` 数组。**不解析 DOM**（洛谷 DOM 变更频繁，JSON 相对稳定） |
| 抓取内容 | AC 题目（`pid` + `name` + `difficulty` + `type`，从 practice 页面 `data.passed` 一次拿全，**不逐题请求详情页**） |
| 难度映射 | 0=暂无评定，1=入门，2=普及-，3=普及/提高-，4=普及+/提高，5=提高+/省选-，6=省选/NOI-，7=NOI/NOI+（实测字段类型为 number，0-7） |
| 一致性校验 | `len(data.passed) === data.user.passedProblemCount`，不等则告警"洛谷返回数据不完整" |
| 双路兼容 | 提取器先找 `lentille-context`（新架构），找不到回退 `window.__INITIAL_STATE__ = decodeURIComponent("...")`（旧架构），防洛谷回滚 |
| 触发方式 | 手动点"更新"按钮，按需触发，无固定周期 |
| 更新粒度 | 单个学生 / 整个班级 |
| 防封策略 | 班级更新时，每人间隔 **2 秒**（实测 2s 连发 20 次未触发风控，可调 1-5s，默认 2s） |
| 失败重试 | 单学生失败退避 10 秒重试，最多 2 次 |
| 部分失败 | **best-effort 模式**：单个学生失败不影响后续；更新结束后 UI 列出失败学生清单 + 单独重试按钮；进度条显示「17/20 成功，3 失败」 |
| 风控识别 | 响应非 200 / 含"登录"/"验证"/"block"/"频繁"关键词 / `lentille-context` 缺失 / 响应大小偏离基线 >30% → 标记"触发风控，暂停 5 分钟"，UI 红字提示 |
| 0 AC 学生 | `data.passed` 为空数组时正常入库，`acRecords` 表对应 0 条记录，`lastSyncedAt` 正常更新 |
| 登录态 | **不需要**（预研实测：未登录可拿完整 AC 列表，`data.passed` 206 条与 `passedProblemCount` 一致） |

> **预研验证**：详见 `预研/01-页面结构探测.md`（字段结构）、`预研/02-频率限制实测.md`（风控实测）
>
> **难度标签用途**：v1 仅采集存储，v2+ 用于实力分布分析。
>
> **字段不含**：`data.passed[i]` 只含 `type/name/difficulty/pid`，**不含**题目标签、AC 时间、提交 ID。如需这些要额外调接口。

### 3.4 UI 注入（核心交互）

> **说明**：列表页徽章与详情页状态条**同时启用**，是互补关系而非二选一。徽章解决"扫读"（列表页快速看哪些题命中），状态条解决"确认"（详情页出题前最后一道关）。

#### 3.4.1 被动触发模型
插件**不主动扫描**任何页面。只有当老师在 Popup 里选了「学生」或「班级」后，注入逻辑才激活。切换选择 = 切换标注内容。未选择时，所有页面保持原生状态，零干扰。

**`currentSelection` 持久化**：
- 存 `chrome.storage.local`，跨标签页跨会话保持
- 通过 `chrome.storage.onChanged` 事件广播到所有标签页的 content script
- 一个标签页切换选择，所有标签页同步更新徽章
- 浏览器重启后默认恢复上次选择
- Popup 提供"清除选择"按钮回到零干扰状态

#### 3.4.2 列表页徽章
**触发位置与选择器策略**（分页面定义，修复 v1"卡片右侧注入"事实性错误）：

| 页面 | URL 模式 | DOM 结构 | 注入策略 |
|---|---|---|---|
| 题单页 | `/training/{id}` | 卡片网格 `.training-problem-card` | 卡片右下角注入 `<span>` 徽章 |
| 题目列表 | `/problem/list` | 表格行 `tr[data-pid]` | 行末尾追加 `<td>` 含徽章 |
| 题目集 | `/problemset` | 表格行 `tr[data-pid]`（列结构不同） | 同上，选择器独立配置 |

**所有选择器集中到 `src/content/selectors.ts`**，洛谷改版时改一处。

**展示形式**：
- 选单个学生：`✅ 张三已AC` 或 灰色「未做过」
- 选班级：`✅ 3/20 已AC`（数字比例）
- hover 徽章：tooltip 显示完整学生名单 + "数据更新于 X 天前"
- **缓存过期提示**：`lastSyncedAt` 超过 7 天（可配）→ 徽章颜色褪色 50% + 加时钟图标

**防重复注入**：注入前 `querySelector('[data-luogu-plus-badge]')` 检查，已注入则更新数据不重复创建。

**视觉规格**（落地参数，修复 v1"玻璃质感"主观描述）：
```css
.luogu-plus-badge {
  padding: 2px 8px;
  border-radius: 4px;
  font-size: 12px;
  backdrop-filter: blur(4px);
  background: rgba(82, 196, 26, 0.15);
  color: #52c41a;
  border: 1px solid rgba(82, 196, 26, 0.3);
}
/* 暗色模式（洛谷暗色背景 #1f1f1f）*/
.luogu-plus-badge[data-theme="dark"] {
  background: rgba(82, 196, 26, 0.25);
  color: #73d13d;
}
/* 过期褪色 */
.luogu-plus-badge[data-stale="true"] {
  opacity: 0.5;
}
```

#### 3.4.3 详情页状态条
**触发位置**：题目详情页 `/problem/{pid}`

**展示形式**：
- 顶部插入状态条
- 选班级：`本班 5/20 已AC：张三、李四、王五、赵六、孙七（数据更新于 2 天前）`
- 选学生：`张三 已 AC` 或 `张三 未做过`
- 关闭语义：**关闭后当前题目页面本次浏览期间不再显示，刷新或重新打开恢复显示**（修复 v1"会话"定义不清）
- a11y：`role="status" aria-live="polite"`

#### 3.4.4 SPA 路由监听
洛谷是 Vue SPA，`pushState`/`replaceState` 切页不会重新加载页面，content script 仅 `document_idle` 注入一次。**必须监听路由变化**：

- `MutationObserver` 监听 `document.body` 子树变化
- 拦截 `history.pushState` / `history.replaceState` + 监听 `popstate`
- 封装 `onRouteChange(callback)` 工具，回调里判断当前页面类型并触发对应注入逻辑
- 路由变化后延迟 300ms 再注入（等 Vue 渲染完成）

### 3.5 管理界面

#### Popup（点击扩展图标）—— v1 唯一入口
- 顶部：当前选中（学生/班级）切换器 + "清除选择"按钮
- 中部：快捷操作（更新当前选中、跳转管理页）
- 底部：链接到完整管理页
- **更新进度反馈**：
  - 更新任务跑在 background，Popup 关闭不影响
  - Popup 重新打开时读 background 任务状态恢复进度条
  - 完成后发 `chrome.notifications` 系统通知 + 扩展图标 badge 文字"✓"
  - 管理页班级列表实时显示"更新中 5/20"

#### 管理页（独立 Tab）
- **学生 Tab**：增删改查、批量导入（见 §3.6）、单点更新、刷新昵称
- **班级 Tab**：增删改查、成员管理、班级批量更新
- **设置 Tab**：请求间隔配置（默认2s，可调1-5s）、缓存过期阈值（默认7天）、主题切换、导出诊断日志
- **关于 Tab**：版本号、开源地址、隐私政策

### 3.6 批量导入流程
- **输入格式**：文本框，一行一个洛谷ID（纯数字）
- **校验**：逐行校验数字格式，非数字行高亮标红
- **抓取**：串行抓取昵称，间隔 2s（与单学生更新一致）
- **去重**：已存在的学生自动跳过，UI 提示"X 个已跳过"
- **失败处理**：失败的行高亮可单独重试，不影响其他行
- **进度**：显示"5/20 已导入"

## 4. 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 框架 | React 18 + TypeScript | 沿用黄sir 经验，类型安全 |
| 构建 | Vite 5 + @crxjs/vite-plugin | CRXJS 是 MV3 生态最成熟的 Vite 插件 |
| 样式 | Tailwind CSS 3 + CSS Modules（注入洛谷页面用） | Tailwind 用于管理页，CSS Modules 隔离注入样式 |
| 状态 | Zustand（各上下文内 UI 状态） | 轻量 |
| 跨上下文同步 | `chrome.storage.local` + `chrome.storage.onChanged` | **修正 v1 误导**：Zustand 本身不跨 content/popup/background，需配合 storage 广播。封装 `useSyncedStore` hook |
| 存储 | IndexedDB（Dexie.js 封装） | 缓存量大，支持 schema 版本升级 |
| Manifest | V3 | Chrome 强制要求 |
| 主题 | 浅色/深色/跟随系统 | 必备，沿用洛谷暗色风格 |
| 依赖体积预算 | React+ReactDOM ~140KB / Dexie ~80KB / Zustand ~10KB / Tailwind purge 后 ~30KB | 预估 ~1.5MB，发版前用 `vite-bundle-visualizer` 验证 |

**预估包体**：~1.5MB（远低于 25MB 上限）

### 4.1 MV3 Service Worker 保活方案（预研验证通过）
**问题**：MV3 SW 默认 30 秒无活动就休眠，班级更新 20 人 × 2s = 40s 长任务必然被中断。

**选定方案**：**chrome.alarms 心跳保活 + IndexedDB 断点续传双保险**
- 更新任务启动时创建 alarm，每 24 秒触发一次（< 30s 休眠阈值）
- alarm 回调里 `chrome.runtime.getPlatformInfo()` 保持 SW 活跃
- 任务完成后清除 alarm
- **双保险**：每抓完一个学生就写 IndexedDB 进度（`taskProgress` 表），SW 万一被杀，下次启动从断点续传

**预研验证**：原型已构建（见 `预研/sw-keepalive-prototype/`），代码可直接迁移主项目 `src/background/`。理论可行性确认，待用户手动加载扩展做最终验证。

**备选方案**：Offscreen Document（Chrome 109+），仅当 alarms 方案手动验证失败才切。

### 4.2 Manifest 权限清单（最小化）
```json
{
  "permissions": ["storage", "notifications", "alarms", "offscreen"],
  "host_permissions": ["*://www.luogu.com.cn/*"]
}
```
**不申请**：`tabs`、`<all_urls>`、`history`、`cookies` 等敏感权限。

### 4.3 浏览器兼容
- **v1 仅支持 Chromium 系**：Chrome 114+ / Edge 114+（`chrome.alarms` + Side Panel API 依赖 114+，虽然 v1 不做 Side Panel 但留版本余量）
- 不支持 Firefox / Safari

## 5. 数据模型

### 5.1 学生表 `students`
```ts
type StudentStatus = 'active' | 'invalid';

interface Student {
  id: string;              // 内部 UUID
  luoguId: number;         // 洛谷 UID，全局唯一索引
  luoguName: string;       // 洛谷昵称，更新时顺带刷新
  remark: string;          // 老师备注名（真实姓名），可留空
  status: StudentStatus;   // active=正常, invalid=洛谷账号失效
  lastSyncedAt: number | null;  // 最近一次成功更新 AC 记录的时间戳，null=从未更新
  createdAt: number;
  updatedAt: number;
}
// 索引：id (主键), luoguId (唯一)
```

### 5.2 班级表 `classes`
```ts
interface Class {
  id: string;              // 内部 UUID
  name: string;            // 班级名
  lastSyncedAt: number | null;  // 班级内所有学生 lastSyncedAt 的最小值，null=有人从未更新
  createdAt: number;
  updatedAt: number;
}
// 索引：id (主键)
// lastSyncedAt 为计算字段，更新班级任一学生时同步刷新
```

### 5.3 班级-学生关系表 `class_members`（替代 v1 的 studentIds 数组）
```ts
interface ClassMember {
  classId: string;         // 班级 ID
  studentId: string;       // 学生 ID
  addedAt: number;
}
// 主键：[classId, studentId] 复合
// 索引：classId, studentId（双向查询）
```
**优势**：删除学生时 `DELETE FROM class_members WHERE studentId=?` 一句话搞定，不用遍历 classes 数组。

### 5.4 AC 记录表 `acRecords`
```ts
interface AcRecord {
  studentId: string;       // 学生内部 ID
  problemId: string;       // 题目 ID，如 "P1234"，外键到 problems 表
  fetchedAt: number;       // 拉取时间戳
}
// 主键：[studentId, problemId] 复合
// 索引：studentId, problemId
// 难度不在此表，移至 problems 表统一管理（见 5.5）
```

### 5.5 题目表 `problems`（新增，解决难度变化同步问题）
```ts
type Difficulty = 0 | 1 | 2 | 3 | 4 | 5 | 6 | 7;
// 0=暂无评定, 1=入门, 2=普及-, 3=普及/提高-, 4=普及+/提高,
// 5=提高+/省选-, 6=省选/NOI-, 7=NOI/NOI+

interface Problem {
  id: string;              // 题目 ID，如 "P1234"/"B2001"，主键（保留 P/B 前缀，原样存储）
  difficulty: Difficulty;  // 最新已知难度（实测 number 类型 0-7）
  title: string;           // 题目标题（已含 [NOIP ...] 等前缀，无需拼接）
  type: string;            // 题目类型 "P"/"B"（历史可能有 T 等，不假设枚举封闭）
  updatedAt: number;       // 难度最后刷新时间
}
// 索引：id (主键), difficulty
// 每次抓取学生 AC 记录时，顺带 upsert 该题难度（洛谷改难度自动同步）
```

## 6. 请求与防封策略

| 场景 | 策略 |
|---|---|
| 单学生更新 | 1 次请求，无间隔 |
| 班级更新 | 每人间隔 2 秒（可调 1-5s，默认 2s） |
| 失败重试 | 退避 10 秒，最多重试 2 次 |
| 部分失败 | best-effort，失败不影响后续，结束时列出失败清单 |
| UA | 浏览器默认 UA，不伪装 |
| 并发 | 串行，不开并发请求 |
| 用户感知 | UI 显示进度条「3/20 已更新」，Popup 关闭后 background 继续跑，完成发系统通知 |
| 风控检测 | 响应含"登录"/"验证"/"block"或 `__INITIAL_STATE__` 缺失 → 暂停 5 分钟，UI 红字提示 |
| 缓存控制 | 配合 `Cache-Control` 头，避免短时间重复抓同一学生 |

**默认配置表**（集中管理，修复 v1 散落多处）：

| 配置项 | 默认值 | 可调范围 |
|---|---|---|
| `requestInterval` | 2000ms | 1000-5000ms |
| `staleThresholdDays` | 7 | 1-30 |
| `retryBackoffMs` | 10000 | 固定 |
| `maxRetry` | 2 | 固定 |
| `rateLimitPauseMs` | 300000（5分钟） | 固定 |

## 12. 合规说明

### 12.1 分发方式
**开源项目 + 老师内部分发**，不上架 Chrome Web Store。理由：
- 绕开商店审核对第三方数据抓取的严格限制
- 目标用户是信息学老师，内部分发够用
- 开源便于社区贡献和审计

### 12.2 洛谷 ToS 评估（待预研确认）
- **robots.txt**：开发前抓取 `luogu.com.cn/robots.txt` 确认 `/user/*/practice` 是否被 Disallow
- **用户协议**：阅读洛谷用户协议"禁止自动化访问"相关条款
- **抓取策略**（若 ToS 灰色）：
  - 用户手动触发，非自动巡检
  - 低频（间隔 2s，单次任务 ≤ 20 请求）
  - 仅存储衍生数据（题目ID + 难度），不存储原文
  - 不商业化、不二次分发数据
- **预研结论将写入附录**

### 12.3 数据隐私
- **数据范围**：学生洛谷ID + 洛谷昵称 + 老师备注名 + AC 题目记录 + 题目难度
- **存储位置**：全部本地 IndexedDB，不上传任何服务器
- **未成年人保护**（PIPL）：备注名允许留空或用化名；扩展卸载时 IndexedDB 自动清除
- **隐私政策**：管理页"关于"Tab 展示，明确"数据仅本地，开发者不接触"

### 12.4 风险承担
- 用户自行承担洛谷封号风险（UI 首次使用提示）
- 默认 2s 间隔为保守值，用户可调至 5s 进一步降风险

## 15. 预研任务（开发前置项）

### ✅ 15.1 洛谷 practice 页面结构探测（已完成）
- **结论**：洛谷已迁移到新架构 "Spilopelia"，数据载体从 `window.__INITIAL_STATE__` 变为 `<script id="lentille-context" type="application/json">` JSON
- **AC 列表路径**：`data.passed` 数组（非 `passedProblems`），每条 `{type, name, difficulty, pid}`
- **难度字段**：`difficulty` 为 number 类型 0-7，从 practice 页面一次拿全
- **未登录可访问**：无 cookie 请求得到 `data.passed` 206 条，与 `data.user.passedProblemCount` 完全一致
- **一致性校验**：`len(data.passed) === data.user.passedProblemCount`
- **字段不含**：题目标签、AC 时间、提交 ID（如需要额外调接口）
- **风险**：架构可能再次变更，提取器需双路兼容（lentille-context 优先，__INITIAL_STATE__ 回退）
- **产物**：`预研/01-页面结构探测.md` + `预研/fixtures/practice-state.json` + `预研/fetch_practice.py`

### ✅ 15.2 未登录访问频率限制实测（已完成）
- **结论**：2s 间隔连发 20 次未触发风控，全部 200，响应稳定 24228 字节，耗时 0.28-0.36s
- **3s 间隔同样安全**，但 2s 已足够安全
- **推荐默认间隔**：**2 秒**（实测确认，覆盖 20 学生上限场景）
- **保留 3s 作为保守降级档**：若上线后偶发风控可一键切换
- **风控特征**（本次未观测到，预期）：非 200 状态码 / "验证"/"频繁"关键词 / 响应大小偏离基线 / 重定向到验证码页
- **风控恢复策略**：指数退避 30s→60s→120s，触发后冷却 5 分钟（C3VK cookie 有效期 5 分钟）
- **测试局限**：①同 URL 重复请求（CDN 命中率高，实际是不同学生页面）②未登录态（实际用老师登录态可能更严）③建议上线后补测
- **产物**：`预研/02-频率限制实测.md` + `预研/fixtures/rate-limit-test.py` + `预研/fixtures/rate-limit-results.json`

### ✅ 15.3 MV3 SW 保活原型验证（已完成）
- **结论**：chrome.alarms 24s 心跳 + storage.local 断点续传 + SW 重启顶层 bootstrap = 双保险，理论可行
- **原型已构建**：4 个文件（manifest.json / background.js / popup.html / popup.js），可直接"加载已解压扩展"运行
- **代码可复用**：原型 → 主项目 `src/background/` 迁移映射已在 `预研/03-SW保活验证.md` 第七章给出
- **待用户手动验证**：加载扩展 → 启动任务 → 关闭 popup → 30s 后查 SW 状态 → 40s 任务完成 → 手动 kill SW 测断点续传
- **备选方案**：Offscreen Document（仅当 alarms 验证失败才切，切换成本：加 offscreen 权限 + 新增 offscreen.html/js + 多一层消息中转）
- **产物**：`预研/sw-keepalive-prototype/` + `预研/03-SW保活验证.md`
